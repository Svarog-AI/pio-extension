# Glossary

## Terms

- **Capability** ‚Äî An AI-driven workflow unit (e.g., `create-goal`, `execute-task`) implemented as a directory package under `src/capabilities/<name>/`. Exposes both a tool (agent-callable) and optionally a command (user-callable via `/pio-*`). Each has a `CapabilityPackageConfig` (default export from `config.ts`) defining its session shape. Non-AI capabilities are consolidated in `direct-tools.ts`.
- **Capability Package** ‚Äî Directory-based capability structure: `config.ts` (CapabilityPackageConfig + register), `role.md`, `workflow.ts`, `guidelines.md`, optional `callbacks.ts` and `schemas.ts`. Replaces the old single-file pattern (`CAPABILITY_CONFIG: StaticCapabilityConfig` in a `.ts` file). Auto-discovered by `discoverCapabilities()`.
- **CapabilityPackageConfig** ‚Äî The config type for directory package capabilities. Defined in `capability-package.ts`. Fields: `name`, `skills`, `validation`, `readOnlyFiles`, `writeAllowlist`, `prepareSession`, `postValidate`, `postExecute`, `frontmatterSchemas` (optional), `inputValidation` (optional ‚Äî declarative file requirements), `preValidate` (optional ‚Äî custom pre-launch callback). Default-exported from `config.ts`.
- **Direct Tools** ‚Äî Non-AI capabilities consolidated in `src/direct-tools.ts`: init, delete-goal, list-goals, parent, create-issue, goal-from-issue, pio_transition. Have no prompts, no sub-sessions, no validation hooks. Exported as `setupDirectTools(pi)`.
- **Goal workspace** ‚Äî A directory under `.pio/goals/<name>/` containing all artifacts for a single feature/fix: `GOAL.md`, `PLAN.md`, step folders (`S01/`, `S02/`), and state markers.
- **Step folder** ‚Äî A zero-padded directory inside a goal workspace (`S01/`, `S02/`) containing the specification (`TASK.md`), implementation artifacts (`TEST.md`, `SUMMARY.md`), review output (`REVIEW.md`), and optionally a `subgoals/` subdirectory for nested subgoal workspaces.
- **prepareSession hook** ‚Äî A `PrepareSessionCallback` defined on `CapabilityPackageConfig.prepareSession` that runs during the `resources_discover` phase, before `before_agent_start`. Used by capabilities to enrich runtime config at startup ‚Äî e.g., `execute-task` and `review-task` read per-step skills from TASK.md frontmatter and merge them into config via `mergeCapabilitySkills()` from `capability-utils.ts`.
- **Sub-session** ‚Äî An isolated pi agent session spawned by `launchCapability()`. Contains a custom `pio-config` entry with prompt, working directory, validation rules, and file protections. Runs one capability per session.
- **Transition** ‚Äî The resolution of "what runs next" after a capability completes. Handled by the declarative state machine framework (`state-machines.ts`): edges with `resolve(context, params) ‚Üí TransitionResult | undefined` replace imperative switch statements. `dispatch()` iterates outgoing edges and collects all matches.
- **StateMachine** ‚Äî Generic type `StateMachine<C>` from `state-machines.ts`. Named config with `id`, `name`, `description`, and `edges: TransitionEdge<C>[]`. The default `goalDrivenDevelopment` machine exports 11 edges covering the pio workflow. Registers explicitly via `setupPioWorkflowMachine()` called from `index.ts`. Machine registry backed by `_machinesById` Map with O(1) lookup via `getMachine(id)`.
- **ResolverResult** ‚Äî Type `Omit<TransitionResult, "stateMachineId">` from `state-machines.ts`. The return type for edge resolver functions ‚Äî `stateMachineId` is omitted because `dispatch()` auto-injects it. Introduced to shift the responsibility for machine ID from individual resolvers to the framework.
- **TransitionEdge** ‚Äî Generic type `TransitionEdge<C>` from `state-machines.ts`. Directed edge `{ from, to, resolve }` where `resolve` is a function combining condition check + param computation. Returns `ResolverResult` when firing, `undefined` otherwise.
- **TransitionResult** ‚Äî Type `{ capability, stateMachineId, params? }` identifying which machine produced the result and which capability should run next. `stateMachineId` is guaranteed by `dispatch()` (auto-injected via `{ ...result, stateMachineId: m.id }`), not set by resolvers. Used by mark-complete for auto-transition routing.
- **Declarative State Machine Framework** ‚Äî The transition system in `state-machines.ts` (replaced `state-machine.ts`). Iteration-based dispatch with no switch statements. Supports explicit machine dispatch (`getMachine(id)` + `dispatch(machine, ...)`) and fallback multi-machine dispatch via `dispatch(undefined, ...)`. Mark-complete reads `stateMachineId` from session params to dispatch against the correct machine. The `stateMachineId` is propagated at top level in enqueued task params.
- **pio_transition Tool** ‚Äî Non-AI tool registered in `direct-tools.ts`. Manual override for transitions: enqueues a task and records the transition audit entry. Used when auto-resolution is undesirable or multiple matches exist.
- **GoalState** ‚Äî A lazy-evaluated filesystem view over a goal workspace (`goal-state.ts`). All methods read fresh from disk ‚Äî no caching. Provides `hasGoal()`, `steps()`, `currentStepNumber()`, etc.
- **Validation gate** ‚Äî The `pio_mark_complete` tool that verifies expected output files exist before allowing session completion. For `review-code`, it also parses YAML frontmatter and creates APPROVED/REJECTED markers automatically.
- **File protection** ‚Äî Two mechanisms enforced via the `tool_call` event handler:
  - *Read-only files:* Certain files (e.g., TASK.md, TEST.md during execution) cannot be modified
  - *Write allowlist:* Only explicitly allowed paths may be written to; blocks all writes to `.pio/` outside the session's goal workspace
- **Session queue** ‚Äî Per-goal task slots at `.pio/session-queue/task-{goalName}.json`. One pending task per goal. Files are consumed (deleted) when launched by `/pio-next-task`.
- **Agent refinement loop detection** ‚Äî A session-guard feature that tracks turn count during pio sessions. When `turnCount` reaches the configured `turnThreshold` (default: 15), it sends a one-time nudge via `pi.sendUserMessage({ deliverAs: "steer" })` encouraging self-diagnosis. The counter resets after each nudge, enabling periodic reminders. Configurable via `guards.turnThreshold` in `~/.pi/pio-config.yaml`. Uses `"steer"` so the nudge interrupts the agent after the current turn's tool calls (mid-run) rather than waiting for the full run to complete.
- **Model config** ‚Äî Optional `~/.pi/pio-config.yaml` that overrides which LLM model a capability uses. Resolution: per-capability ‚Üí default ‚Üí inherit parent.

- **Finalize Goal Agent** ‚Äî The agent that runs after a goal is fully completed. Reads accumulated decisions from DECISIONS.md, per-step SUMMARY.md files, and PLAN.md, then evaluates each finding against update rules to determine which `.pio/PROJECT/*.md` files need updates.
- **CapabilitySkills** ‚Äî The config shape for declaring skills per-capability: `{ mandatory?: string[], recommended?: { name: string; condition: string }[] }`. Both fields optional. Defined in `src/types.ts`. Mirrored structurally by `TASK_FRONTMATTER_SCHEMA` in capability-local schemas.
- **Auto-discovery** ‚Äî `discoverCapabilities(baseDir)` scans `src/capabilities/` for directories containing `config.ts`, dynamically imports configs, and returns `CapabilityPackageDescriptor[]`. Test fixtures (`test-*` prefixed) are filtered out. Replaces hardcoded per-capability imports in `index.ts`.
- **Prompt Compiler** ‚Äî `prompt-compiler.ts` exports `compilePrompt()` which reads component files from capability package directories (`role.md`, `workflow.ts`, `guidelines.md`) and assembles the final prompt. Also exports `readWorkflowSteps()`, `renderWorkflowSection()`, `mergeWorkflowStepSkills()`, `readPackageComponents()`. Strict leaf module.
- **Step Nudging** ‚Äî `guards/step-nudging.ts` implements a `workflow-step-finish` tool that agents call to advance through multi-step workflows. State tracked via `resources_discover` (reads `sessionParams.workflowSteps`). Nudge injected via `turn_end` + `pi.sendMessage({ deliverAs: "steer" })`. Default behavior (no tool call) means staying on current step.
- **Frontmatter Schema Validation** ‚Äî Capabilities declare `frontmatterSchemas: FrontmatterSchemaDeclaration[]` on `CapabilityPackageConfig`. The exit-gate in `guards/validation.ts` validates output YAML against TypeBox schemas before allowing session completion. Schemas defined in capability-local `schemas.ts` files (replaced shared `src/frontmatter-schemas.ts`).
- **Callback File** ‚Äî Convention: lifecycle callbacks (validation, file protection resolvers) live in `callbacks.ts` inside capability directories. Previously named `validators.ts` until Step 19 convention cleanup renamed all instances.
- **Capability Utils** ‚Äî `capability-utils.ts`: leaf utility exporting `mergeCapabilitySkills(base, additional)` accepting `CapabilitySkills` for both parameters. Extracted from `session-capability.ts` to break encapsulation inversion (shared infrastructure depending on capability-local types).
- **Per-Step Skills (TASK.md Frontmatter)** ‚Äî Skills declared in TASK.md YAML frontmatter, read by `StepStatus.taskSkills()` during `prepareSession`. Merged with base capability skills via `mergeCapabilitySkills()` (Set-based dedup for mandatory, Map-based first-seen-wins for recommended). Provides runtime skill resolution beyond static capability configs.
- **Skill Injection** ‚Äî Dynamic skill loading at session startup via `buildSkillLoadingSection()` in `capability-session.ts`. Resolution order: per-step skills (TASK.md frontmatter or workflow.ts step declarations) ‚Üí base capability skills (CapabilityPackageConfig.skills) ‚Üí global mandatory skills (`pio`, `ask-user`). Mandatory skills are force-injected (content wrapped in `<skill>` XML tags); recommended skills listed as loading instructions.
- **DECISIONS.md** ‚Äî Accumulated across steps starting at Step 2, documents architectural decisions, plan deviations, and file placement choices made during specification and implementation. Read by the Finalize Goal Agent to update project documentation.
- **Plan Frontmatter** ‚Äî The YAML frontmatter block in PLAN.md containing `totalSteps` and a `steps` array of `{ name, complexity? }` entries. `complexity` is optional, defaulting to `"task"`. Subgoal steps are declared here with `complexity: "subgoal"`. This frontmatter is the single source of truth for step definitions ‚Äî `GoalState.steps()` derives from it rather than scanning folders.
- **Queue Key** ‚Äî The key used to address a goal's task slot in `.pio/session-queue/`. Flat goals use the goal name basename. Nested subgoals produce hierarchical keys (e.g., `parent__S03__nested`) via `deriveQueueKey()`, using `__` as delimiter.
- **Subgoal** ‚Äî A child goal workspace spawned by a plan step with `complexity: "subgoal"`. Lives under `S{NN}/subgoals/<name>/` inside the parent step folder. Runs through the full pio lifecycle recursively (create-goal ‚Üí create-plan ‚Üí evolve-plan ‚Üí execute-task ‚Üí review-code ‚Üí finalize-goal). On completion, propagates back to the parent's evolve-plan.
- **acli (Atlassian CLI)** ‚Äî Command-line tool for interacting with Jira. Used by agents via `bash` tool calls as described in the `pio-jira` skill. Provides issue pull/push, JQL search, and auth management without TypeScript capability code.
- **Jira Integration** ‚Äî A skill-only integration: all Jira operations (auth check, pull tickets to local issues, push local issues to Jira, JQL search) are guided by `src/skills/pio-jira/SKILL.md` rather than TypeScript capabilities. Agents invoke `acli` via bash.
- **Jira config file** ‚Äî Optional `.pio/jira-config.yaml` storing `site`, `projectKey`, and `defaultType`. Created by the `setup-config.sh` script (bundled in `src/skills/pio-jira/scripts/`) before any Jira operation when the config is missing. Used for push, pull, and search operations.
- **Plan Revision** ‚Äî A workflow capability (`revise-plan`) triggered when evolve-plan's specification writer detects significant divergence from the plan. Archives current PLAN.md to `PLAN_ARCHIVE/`, deletes incomplete step folders, and rewrites a fresh plan with completed steps as anchors.
- **REVISE_PLAN_NEEDED** ‚Äî A marker file written by the specification writer (evolve-plan agent) inside a step folder (`S{NN}/REVISE_PLAN_NEEDED`) to signal that the plan must be revised. The transition resolver detects this via `StepStatus.revisionNeeded()` and routes to `revise-plan` instead of continuing normally.
- **PLAN_ARCHIVE** ‚Äî Directory inside a goal workspace (`<goalDir>/PLAN_ARCHIVE/`) storing timestamped copies of PLAN.md before each revision. Enables the revise-plan agent to reference previous plan attempts.

## Business Concepts

- **Goal-driven workflow** ‚Äî The core concept: complex work is structured as goals, each decomposed into a plan of ordered steps. Each step is specified (TASK.md with acceptance criteria), implemented test-first (executor derives tests from TASK.md using TDD), and reviewed (approve/reject) in isolated sub-sessions. Complex steps can be further decomposed via nested subgoals.
- **Specification-driven implementation** ‚Äî `evolve-plan` produces TASK.md (what to build, with acceptance criteria) before any code is written. The executor (`execute-task`) derives test cases from the acceptance criteria using TDD methodology, then writes tests first and implements. This separates planning from execution and enables focused sub-sessions.
- **Subgoal lifecycle** ‚Äî Complex steps marked `complexity: "subgoal"` in plan frontmatter spawn a child goal under `S{NN}/subgoals/<name>/`. The child runs the full pio lifecycle independently. On completion, `finalize-goal` propagates back to the parent's `evolve-plan`, which resumes the parent workflow via the session queue.
- **Review gate** ‚Äî After implementation, `review-code` evaluates quality, test coverage, and alignment with requirements. Approval advances the workflow; rejection sends the step back for re-execution with feedback from REVIEW.md.
- **DECISIONS.md carryover** ‚Äî Starting at Step 2, `evolve-plan` writes DECISIONS.md to document architectural decisions made during specification. Subsequent steps read this to maintain consistency across the implementation.
- **Transition audit trail** ‚Äî Each capability completion records a transition entry in `<goalDir>/transitions.json`: an append-only JSON array tracking from‚Üíto transitions with timestamps and params. Provides observability into the workflow's progress.
- **Finalization gate** ‚Äî After a goal completes, the finalize-goal capability reads accumulated decisions and updates `.pio/PROJECT/*.md` documentation. The pio-project-knowledge skill provides canonical update rules that map decision categories to target files. This ensures future goals benefit from accumulated knowledge.
