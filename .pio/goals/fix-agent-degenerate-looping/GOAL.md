# Fix Agent Degenerate Looping Instead of Tool Use

Agents sometimes enter degenerate loops running bash commands (like `date +%Y%m%d_%H%M%S` and `ls`) instead of calling available `pio_*` tools. This wastes turns, confuses users, and delays actual work. The goal is to eliminate misleading documentation and improve prompt/tool descriptions so agents reliably use pio tools directly.

## Current State

The pio extension exposes workflow tools (`pio_create_issue`, `pio_goal_from_issue`, `pio_create_goal`, etc.) that handle all filesystem operations internally. No bash commands should be needed for pio workflow operations. However, agents exhibit a degenerate pattern: repeatedly running `bash date +%Y%m%d_%H%M%S` and `ls .pio/issues/` instead of calling tools like `pio_create_issue`.

**Root cause — misleading documentation:** Two key files contain an outdated claim that directly instructs the agent to generate timestamps:

- **`src/skills/pio/SKILL.md`** (line 55): States "Queue files use timestamps: Task filenames in `.pio/session-queue/` are `{timestamp}-{capability}.json`. Lexicographic sort = chronological order for FIFO processing."
- **`.pio/PROJECT.md`** (generated by `pio_create_project_context`, mirrors SKILL.md content): Contains the same outdated timestamp claim.

The actual code in `src/utils.ts` (`enqueueTask`) uses per-goal files: `task-{goalName}.json`. No timestamps are involved. When an agent reads SKILL.md or PROJECT.md and sees "Queue files use timestamps", it attempts to generate a timestamp by running `bash date +%Y%m%d_%H%M%S`, creating the observed loop.

**Additional contributing factors:**

- Tool descriptions (e.g., `pio_create_issue` in `src/capabilities/create-issue.ts`) are functional but don't explicitly discourage bash alternatives. The agent doesn't know these tools "handle all filesystem operations internally" unless it's told.
- The SKILL.md overview is strong on workflow mechanics but lacks behavioral guidance like "always use pio_* tools — never manually create files in `.pio/` via bash or the write tool."

## To-Be State

**1. Remove misleading timestamp documentation.** The claim about `{timestamp}-{capability}.json` queue files must be removed from `src/skills/pio/SKILL.md` (and consequently won't propagate to `.pio/PROJECT.md` after regeneration). Replace with an accurate description: "Task files use per-goal slots (`task-{goalName}.json`) — one pending task per goal."

**2. Add explicit behavioral guidance to SKILL.md.** Include a section or callout instructing agents to always use `pio_*` tools for workflow operations. This should make clear that:
- Tools handle all filesystem operations internally
- Manual bash commands (`ls`, `date`, etc.) are unnecessary for pio workflow tasks
- The write tool should not be used to create files in `.pio/` when a pio tool exists for that purpose

**3. Improve tool descriptions.** Update tool `description` fields (in `defineTool` calls) to be more actionable and self-contained. For example, `pio_create_issue` should read: "Create a new issue as a markdown file under `.pio/issues/`. Use this tool directly — no bash commands or manual file creation needed."

**4. Ensure SKILL.md stays current with the codebase.** Update any other stale references in `src/skills/pio/SKILL.md` (e.g., repository structure section, command reference table) so agents aren't misled by outdated information.

The net effect: when an agent reads pio documentation (SKILL.md injected as a skill, PROJECT.md injected as project context), it will not be instructed to generate timestamps, and it will have explicit guidance to use tools directly instead of bash workarounds.
