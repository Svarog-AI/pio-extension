# Plan: Implement Nested Subgoals

Add nested subgoal support to pio: plan steps marked `[subgoal]` spawn child goal workspaces under `S{NN}/subgoals/<name>/` that run through the full pio lifecycle recursively. All changes are additive — existing flat goals and plans function identically.

## Prerequisites

None.

## Steps

## Step 1: Path resolution infrastructure

Extend path resolution to support nested subgoal directories. `resolveGoalDir` gains an optional `parentStepDir` parameter; when present, it resolves `path.join(parentStepDir, "subgoals", name)` instead of `<cwd>/.pio/goals/<name>`. Existing callers without this parameter work identically.

`deriveSessionName` is extended to format qualified names for display: when a goal name contains `__` delimiters (from hierarchical queue keys), replace them with `/` in the session display name so subgoal sessions appear as `parent/S03/nested execute-task s1` instead of `parent__S03__nested execute-task s1`.

**Acceptance Criteria**
- `npx tsc --noEmit` reports no errors
- Running existing test suite (`npm test`) passes with no regressions
- `resolveGoalDir(cwd, "my-feature")` returns the same flat path as before (backward compatible)
- `resolveGoalDir(cwd, "nested", parentStepDir)` returns `<parentStepDir>/subgoals/nested`

**Files Affected**
- `src/fs-utils.ts` — add optional `parentStepDir` param to `resolveGoalDir`; update `deriveSessionName` formatting for `__` → `/`
- `src/fs-utils.test.ts` — tests for nested path resolution and backward compatibility

## Step 2: Queue keying

Introduce hierarchical queue keys to prevent collisions between sibling subgoals and flat goals. New pure function `deriveQueueKey(goalDir, cwd)` strips the `.pio/goals/` prefix from the goal directory path, filters out `subgoals` directory markers, and joins remaining segments with `__`. Flat goals produce identical output: `my-feature` → `my-feature`. Nested goals produce qualified keys: `parent/S03/subgoals/nested` → `parent__S03__nested`.

`enqueueTask` and `readPendingTask` accept optional `qualifiedName` parameter. When present, use it as the queue filename key instead of deriving from goal basename. `GoalState.pendingTask()` computes the qualified queue key from `goalDir` using `deriveQueueKey`. `listPendingGoals` returns qualified names (may contain `__` delimiters).

**Acceptance Criteria**
- `npx tsc --noEmit` reports no errors
- Running existing test suite passes with no regressions
- `deriveQueueKey("/repo/.pio/goals/my-feature", "/repo")` returns `"my-feature"` (flat goal unchanged)
- `deriveQueueKey("/repo/.pio/goals/parent/S03/subgoals/nested", "/repo")` returns `"parent__S03__nested"`
- Queue file for flat goal is still `task-my-feature.json` (backward compatible)

**Files Affected**
- `src/queues.ts` — add `deriveQueueKey()`; extend `enqueueTask`, `readPendingTask` with optional `qualifiedName` param; update `listPendingGoals` return type docs
- `src/goal-state.ts` — `pendingTask()` computes qualified queue key via `deriveQueueKey` instead of using `goalName` basename
- `src/queues.test.ts` — tests for `deriveQueueKey` and hierarchical queue keys

## Step 3: State machine transitions

Add subgoal spawning and completion propagation to the state machine. All new params (`parentGoalName`, `parentStepNumber`, `subgoalType`) are optional — callers without them see identical behavior.

**Subgoal spawning** in `transitionEvolvePlan`: When the current step is flagged as a subgoal (detected via `[subgoal]` annotation in PLAN.md step headings), route to `create-goal` instead of `execute-task`, with `parentGoalName`, `parentStepNumber`, `subgoalType: true`, and **explicit `workingDir`** in params. The `workingDir` is the full nested path: `<cwd>/.pio/goals/<parent>/S{NN}/subgoals/<name>`. This is critical — without explicit `workingDir`, `capability-config.ts` falls back to flat `resolveGoalDir(cwd, "nested")` which produces a wrong directory. New helper `getStepAnnotation(goalDir, stepNumber)` reads PLAN.md body using an extended heading regex and checks for the `[subgoal]` marker (or `[subgoal: name]`) in the heading for the given step number.

**Completion propagation**: Extract `finalize-goal` handling into a new `transitionFinalizeGoal` function. For subgoals (has `parentGoalName` param), returns `evolve-plan` for the parent with `stepNumber: parentStepNumber + 1`. Does NOT forward `parentGoalName` or `parentStepNumber` to prevent param pollution. For top-level goals, returns `undefined` (terminal, unchanged behavior).

Update `resolveTransition`: wire up `transitionFinalizeGoal` as the handler for the `finalize-goal` case instead of the inline `undefined` return.

**Acceptance Criteria**
- `npx tsc --noEmit` reports no errors
- Running existing test suite passes with no regressions
- `resolveTransition("evolve-plan", state, { goalName: "parent", stepNumber: 3 })` routes to `create-goal` when step 3 has `[subgoal]` annotation
- Subgoal spawning params include explicit `workingDir` (full nested path), `parentGoalName`, `parentStepNumber`, and `subgoalType: true`
- `resolveTransition("evolve-plan", state, { goalName: "parent", stepNumber: 4 })` routes to `execute-task` when step 4 has no annotation (backward compatible)
- `resolveTransition("finalize-goal", state, { goalName: "nested", parentGoalName: "parent", parentStepNumber: 3 })` routes to `evolve-plan` for parent with `stepNumber: 4`
- `transitionFinalizeGoal` does NOT forward `parentGoalName`/`parentStepNumber` in returned params (param pollution prevention)
- `resolveTransition("finalize-goal", state, { goalName: "my-feature" })` returns `undefined` (top-level goal, backward compatible)

**Files Affected**
- `src/state-machine.ts` — add `getStepAnnotation()` helper (reads `[subgoal]`/`[subgoal: name]` from PLAN.md body via extended heading regex); modify `transitionEvolvePlan` for subgoal spawning with explicit `workingDir`; extract `transitionFinalizeGoal`; update `resolveTransition` switch
- `src/state-machine.test.ts` — tests for subgoal spawning, completion propagation, backward-compatible paths

## Step 4: Evolve-plan integration

Detect subgoal steps before launching the specification writer session. When the next step is flagged as `[subgoal]`, skip TASK.md/TEST.md generation and route to `create-goal` with parent context instead.

In `validateAndFindNextStep`: After finding the next step number, check if it's a subgoal step using `getStepAnnotation()` from the state machine. If so, return a result indicating it's a subgoal (extend return type or add a flag along with the derived subgoal name). The caller in `handleEvolvePlan` then routes to `create-goal` with `parentGoalName`, `parentStepNumber`, and explicit `workingDir` for the subgoal workspace (`<stepDir>/subgoals/<name>`).

Update the tool's `execute` method similarly: when a subgoal is detected, enqueue a `create-goal` task instead of an `evolve-plan` task.

**Acceptance Criteria**
- `npx tsc --noEmit` reports no errors
- Running existing test suite passes with no regressions
- Non-subgoal steps proceed identically — `validateAndFindNextStep` returns ready state, launches evolve-plan session as before
- Subgoal steps are detected and routed to `create-goal` instead of producing TASK.md/TEST.md

**Files Affected**
- `src/capabilities/evolve-plan.ts` — subgoal step detection in `validateAndFindNextStep`; routing in `handleEvolvePlan` and tool execute
- `src/capabilities/evolve-plan.test.ts` — tests for subgoal detection and routing

## Step 5: Session capability + create-plan integration

Two independent changes to close the loop on subgoal lifecycle.

**Session capability:** In `pio_mark_complete`, when enqueuing the next task, use the transition's `params.goalName` (not `state.goalName`) as the queue key goal name. This enables parent queue slot restoration when a subgoal completes — the transition for subgoals returns the parent's goal name in params, so enqueuing to the parent's slot allows `/pio-next-task` to resume the parent workflow.

**Create-plan validation:** Extend `postValidateCreatePlan` to parse `[subgoal]` annotations from step headings and verify they are consistent. Count headings with `[subgoal]` suffix and ensure they don't exceed total count. If annotations are found, the plan is valid — no special enforcement needed beyond basic sanity checks. The `STEP_HEADING_RE` should be updated to also capture the optional `[subgoal]` suffix so heading counts remain accurate (a heading like `## Step 3: Build pipeline [subgoal]` must still match).

**Acceptance Criteria**
- `npx tsc --noEmit` reports no errors
- Running existing test suite passes with no regressions
- `pio_mark_complete` uses transition's adjusted `params.goalName` for enqueuing (parent goal name for subgoals)
- `postValidateCreatePlan` accepts plans with `[subgoal]` annotations in step headings without failing

**Files Affected**
- `src/capabilities/session-capability.ts` — use `nextTask.params?.goalName` for enqueuing instead of `state.goalName`
- `src/capabilities/create-plan.ts` — update `STEP_HEADING_RE` to handle `[subgoal]` suffix; extend `postValidateCreatePlan` to validate annotations
- `src/capabilities/session-capability.test.ts` — test for goalName propagation in enqueuing
- `src/capabilities/create-plan.test.ts` — test for annotation parsing and validation

## Step 6: List-goals recursive scan

For each discovered subgoal, show it indented or prefixed with the parent path (e.g., `parent/S03/nested`) so the table displays the full hierarchy. A subgoal's phase is inferred from its own files (`GOAL.md`, `PLAN.md`, step folders) using the same logic as flat goals.

## Step 7: Prompts and skills

Update documentation so agents and users understand the subgoal feature. Five files require changes, all additive content.

**`src/prompts/create-plan.md`:** Add subgoal decomposition instructions. Planning agent evaluates each step against leaf-node criteria (can the step be verified with an I/O contract test? is it self-contained enough to run independently?) and marks composite steps with `[subgoal]` annotations in the heading: `### Step 3: Build the pipeline [subgoal]`. Step count guard (`totalSteps > 8`) prevents flat trees — if a plan would exceed 8 steps, encourage subgoal decomposition.

**`src/prompts/evolve-plan.md`:** Add instructions for the fallback correction path. If the specification writer discovers an unmarked step is actually composite (spec writer has more concrete info than the planner), document how to signal this — however, note that the primary detection happens in `validateAndFindNextStep` (code-level, Step 4) before the evolve-plan session launches. The prompt covers the edge case where the planning agent missed a composite step.

**`src/prompts/finalize-goal.md`:** Add subgoal-aware summary reading. When a step has subgoals (indicated by a `subgoals/` directory under the step folder), read the subgoal's completion state and summaries instead of expecting TASK.md/TEST.md at the step level.

**`src/prompts/create-goal.md`:** Add parent context reading instructions. For subgoal sessions, the initial message (injected by `transitionEvolvePlan`) includes the parent goal directory path. Instruct the Goal Definition Assistant to read the parent's `GOAL.md` and relevant step from `PLAN.md` at that path to understand decomposition scope. **`src/skills/pio-planning/SKILL.md`:** Document leaf-node criteria, `[subgoal]` annotation format, and decomposition guards (step count guard at 8). Add a "Subgoal Decomposition" section to step design rules.

## Notes

- **Backward compatibility is critical.** Every change must be additive. Flat goals without subgoal metadata function identically. Test with a flat goal workflow at each step to verify.
- **Param scoping:** `parentGoalName` and `parentStepNumber` are top-level params on subgoal sessions only. They are checked explicitly, never recursed into `_sessionContext`, and not forwarded to parent's evolve-plan. This prevents param pollution across nesting levels.
- **CWD derivation in goal-state.ts** already works correctly at all nesting depths (uses `indexOf("/goals/")`). No changes needed there beyond queue keying.
- **File protection:** No changes to `validation.ts` — the default-deny check (`tp.startsWith(workingDir + path.sep)`) works for nested paths automatically. The spawning transition MUST pass explicit `params.workingDir` (full nested path) to bypass flat `resolveGoalDir` derivation in `capability-config.ts`.
- **`listPendingGoals` returns qualified names** containing `__` delimiters. Downstream code (e.g., `/pio-list-goals`) must handle hierarchical names — the `resolveGoalDir` extension from Step 1 reconstructs nested paths.
- **No frontmatter schema change.** The `PLAN_FRONTMATTER_SCHEMA` remains unchanged; subgoal detection uses in-body `[subgoal]` annotations only (primary mechanism per SYNTHESIS.md).
- **Subgoal name derivation:** When a step heading has `[subgoal]` without an explicit name, derive the subgoal name by slugifying the step title (e.g., "Implement OAuth flow" → `oauth-flow`). The annotation format supports both `[subgoal]` and `[subgoal: explicit-name]`.

For the condensed summary with recommended decisions, see `docs/plans/subgoals/SYNTHESIS.md`.
