# Plan: Implement Nested Subgoals

Add nested subgoal support to pio: plan steps marked as composite in PLAN.md frontmatter spawn child goal workspaces under `S{NN}/subgoals/<name>/` that run through the full pio lifecycle recursively. All changes are additive — existing flat goals and plans function identically.

## Prerequisites

None.

## Steps

### Step 1: Path resolution infrastructure [COMPLETED]

Extended path resolution to support nested subgoal directories. `resolveGoalDir` gains optional `parentStepDir` parameter for nested paths; `deriveSessionName` replaces `__` with `/` for hierarchical display names. Backward compatible — existing flat-goal callers work identically.

**Status:** COMPLETED — implementation approved in S01, committed (`d15707a`). Do not modify.

### Step 2: Queue keying [COMPLETED]

Introduced `deriveQueueKey(goalDir, cwd)` to produce unique hierarchical queue keys (e.g., `parent__S03__nested`). Extended `enqueueTask` and `readPendingTask` with optional `qualifiedName` parameter. Updated `GoalState.pendingTask()` to compute qualified keys. Flat goals produce identical output — no migration needed.

**Status:** COMPLETED — implementation approved in S02, committed (`8eba42b`). Do not modify.

### Step 3: Mandatory plan frontmatter with per-step metadata

Extend PLAN.md frontmatter schema and validation so every step has structured metadata (complexity and subgoal name). Subgoal detection reads from this frontmatter — no body-scanning of markdown headings.

**New `PLAN_FRONTMATTER_SCHEMA` fields:**
- Required `steps` array: each entry has `number` (integer), optional `subgoalName` (string, present only for composite steps)
- When `subgoalName` is present, that step spawns a subgoal workspace named by that value
- A step without `subgoalName` is a leaf node — normal TASK.md/TEST.md generation

**`postValidateCreatePlan` changes:** Validate that the `steps` array exists, has exactly `totalSteps` entries, each entry has a valid `number`, and `subgoalName` (if present) is a non-empty string. Plans without subgoals have all entries without `subgoalName`. Existing plans missing the `steps` field are treated as having no subgoal metadata (all regular steps) for backward compatibility — but new plans created by `create-plan` must include it.

**TypeBox schema change in `src/frontmatter-schemas.ts`:** Add optional `steps` array to `PLAN_FRONTMATTER_SCHEMA`. Each step entry: `{ number: Integer, subgoalName?: String }`. The field is optional at the schema level (backward compatible) but enforced by `postValidateCreatePlan` for new plans.

**Acceptance criteria:**
- [ ] `npx tsc --noEmit` reports no errors
- [ ] Running existing test suite passes with no regressions
- [ ] Plans without `steps` field validate as valid but have no subgoal metadata (backward compatible)
- [ ] Plans with `steps` array and `subgoalName` entries pass `postValidateCreatePlan` validation
- [ ] `postValidateCreatePlan` rejects plans where `steps` array length doesn't match `totalSteps`
- [ ] `postValidateCreatePlan` rejects plans where step entries have invalid `number` values or empty `subgoalName`

**Files affected:**
- `src/frontmatter-schemas.ts` — extend `PLAN_FRONTMATTER_SCHEMA` with optional `steps` array containing `{ number, subgoalName? }` entries
- `src/capabilities/create-plan.ts` — update `postValidateCreatePlan` to validate `steps` array (length matches `totalSteps`, valid step numbers, non-empty `subgoalName` when present)

### Step 4: State machine transitions

Add subgoal spawning and completion propagation to the state machine. Subgoal detection reads from PLAN.md frontmatter (`steps` array with `subgoalName`), NOT from body-scanning of markdown headings.

**New helper `getStepSubgoal(goalDir, stepNumber)`:** Reads PLAN.md frontmatter using existing `extractFrontmatter()` + `PLAN_FRONTMATTER_SCHEMA` validation. Looks up the step entry matching `stepNumber` in the `steps` array. Returns the `subgoalName` if present, or `null` if the step is a regular leaf node (or if frontmatter parsing fails). No regex heading scanning — purely frontmatter-based.

**Subgoal spawning in `transitionEvolvePlan`:** When the current step's `subgoalName` is non-null (from frontmatter), route to `create-goal` with `parentGoalName`, `parentStepNumber`, `subgoalType: true`, and explicit `workingDir` (full nested path: `<cwd>/.pio/goals/<parent>/S{NN}/subgoals/<name>`). Uses the `resolveGoalDir(cwd, subgoalName, parentStepDir)` extension from Step 1.

**Completion propagation:** Extract `finalize-goal` handling into a new `transitionFinalizeGoal` function. For subgoals (has `parentGoalName` param), returns `evolve-plan` for the parent with `stepNumber: parentStepNumber + 1`. Does NOT forward `parentGoalName` or `parentStepNumber` to prevent param pollution. For top-level goals, returns `undefined` (terminal, unchanged).

**Update `resolveTransition`:** Wire up `transitionFinalizeGoal` as the handler for the `finalize-goal` case instead of the inline `undefined` return.

**Acceptance criteria:**

- [ ] Running existing test suite passes with no regressions

### Step 5: Evolve-plan integration

Detect subgoal steps before launching the specification writer session. When the next step has a `subgoalName` in PLAN.md frontmatter, skip TASK.md/TEST.md generation and route to `create-goal` with parent context instead.

In `validateAndFindNextStep`: After finding the next step number, check if it's a subgoal step using `getStepSubgoal()` from the state machine (reads frontmatter). If so, return a result indicating it's a subgoal along with the subgoal name. The caller in `handleEvolvePlan` then routes to `create-goal` with `parentGoalName`, `parentStepNumber`, and explicit `workingDir` for the subgoal workspace (`<stepDir>/subgoals/<name>`).

Update the tool's `execute` method similarly: when a subgoal is detected, enqueue a `create-goal` task instead of an `evolve-plan` task.

**Acceptance criteria:**

- [ ] Running existing test suite passes with no regressions
- [ ] Non-subgoal steps proceed identically — `validateAndFindNextStep` returns ready state, launches evolve-plan session as before

### Step 6: Session capability + create-plan integration

Two independent changes to close the loop on subgoal lifecycle.

**Session capability:** In `pio_mark_complete`, when enqueuing the next task, use the transition's `params.goalName` (not `state.goalName`) as the queue key goal name. This enables parent queue slot restoration when a subgoal completes — the transition for subgoals returns the parent's goal name in params, so enqueuing to the parent's slot allows `/pio-next-task` to resume the parent workflow.

**Create-plan prompt:** Update `postValidateCreatePlan` to also validate that step entries with `subgoalName` have unique names (no duplicate subgoal names across steps). The `STEP_HEADING_RE` regex should NOT be changed — it already matches all `## Step N:` headings correctly regardless of body content.

## Notes

- **Backward compatibility is critical.** Every change must be additive. Flat goals without subgoal metadata function identically. The `steps` field in frontmatter is optional at the schema level — existing plans validate unchanged but are treated as having no subgoals.
- **No body-scanning for subgoal detection.** Subgoal metadata lives exclusively in PLAN.md frontmatter (`steps` array with `subgoalName`). No regex heading parsing, no `[subgoal]` annotations in markdown body. The planning agent writes the structured frontmatter; downstream code reads it programmatically.
- **Param scoping:** `parentGoalName` and `parentStepNumber` are top-level params on subgoal sessions only. They are checked explicitly, never recursed into `_sessionContext`, and not forwarded to parent's evolve-plan. This prevents param pollution across nesting levels.
- **File protection:** No changes to `validation.ts` — the default-deny check (`tp.startsWith(workingDir + path.sep)`) works for nested paths automatically. The spawning transition MUST pass explicit `params.workingDir` (full nested path) to bypass flat `resolveGoalDir` derivation in `capability-config.ts`.
- **CWD derivation in goal-state.ts** already works correctly at all nesting depths (uses `indexOf("/goals/")`). No changes needed there beyond queue keying.
- **Step count guard:** When `totalSteps > 8`, the planning agent should consider subgoal decomposition. This is enforced by prompt instructions and/or post-validation, not by schema constraints.
- **`listPendingGoals` returns qualified names** containing `__` delimiters. Downstream code (e.g., `/pio-list-goals`) must handle hierarchical names — the `resolveGoalDir` extension from Step 1 reconstructs nested paths.
- **No test file for list-goals.** `src/capabilities/list-goals.ts` has no dedicated test file. The executor should add one if the recursive scan logic warrants unit-level verification.
