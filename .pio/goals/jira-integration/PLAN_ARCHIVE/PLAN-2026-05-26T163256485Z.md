# Plan: Jira Integration via Atlassian CLI (acli)

Add two commands and a skill to bridge Jira tickets with local `.pio/issues/` files using `acli`.

## Prerequisites

- `npm run check` passes with no errors
- `npm test` passes with no regressions

## Steps

### Step 1: Jira Utilities Module

**Description**

Create a shared utility module for all `acli` interaction. This centralizes child process spawning, JSON parsing, error detection, and config file reading so both capabilities reuse the same code. The module exports pure functions that accept `cwd` and return structured results — no tool/command registration here.

Key exports:
- `runAcli(cwd, args: string[]): { stdout, stderr, exitCode }` — spawns `acli` with args, checks availability (`which acli` equivalent), parses JSON output, detects auth errors
- `readJiraConfig(cwd): JiraConfig | undefined` — reads `.pio/jira-config.yaml` using `js-yaml`, returns typed config or undefined if file missing
- `jiraKeyToSlug(key: string): string` — derives issue slug from Jira key (e.g., `PROJ-123` → `jira-proj-123`)

Interface signatures:

```ts
interface JiraConfig {
  projectKey?: string;
  defaultType?: string;
}

interface AcliResult {
  stdout: Record<string, unknown>;
  stderr?: string;
  exitCode: number;
}
```

**Acceptance Criteria**

- `npm run check` reports no errors
- `runAcli` returns an error when `acli` is not found on PATH
- `runAcli` detects "unauthorized" in output and returns a helpful message referencing `acli jira auth login`
- `readJiraConfig` returns `undefined` when config file doesn't exist
- `readJiraConfig` parses valid YAML and returns typed fields
- `jiraKeyToSlug` converts `PROJ-123` to `jira-proj-123`

**Files Affected**

- `src/capabilities/jira-utils.ts` — new file: shared acli utilities

### Step 2: jira-to-issue Capability

**Description**

Implement the `pio_jira_to_issue` tool and `/pio-jira-to-issue` command. Pulls a Jira ticket by key, extracts summary and description from JSON output, and creates a local issue file at `.pio/issues/jira-<project>-<number>.md`. If the issue already exists (slug collision), returns a warning without overwriting.

Uses `runAcli()` to execute `acli jira workitem view KEY --json`, parses the response for `summary` and `description` fields, then delegates to the existing `createIssue()` function from `create-issue.ts` (or writes the file directly following the same pattern).

**Acceptance Criteria**

- `npm run check` reports no errors
- `pio_jira_to_issue` is registered as a tool via `pi.registerTool()`
- `/pio-jira-to-issue` is registered as a command via `pi.registerCommand()`
- Returns an error when `acli` is not installed (delegated to `runAcli`)
- Returns an error when user is not authenticated (detected by `runAcli`)
- Creates `.pio/issues/jira-proj-123.md` with correct slug derived from key
- Returns a warning when the issue file already exists without overwriting

**Files Affected**

- `src/capabilities/create-issue.ts` — export the `createIssue` function (currently private) so it can be reused by jira-to-issue
- `src/capabilities/jira-to-issue.ts` — new file: tool, command, and setup function following create-issue.ts pattern
- `src/index.ts` — import and call `setupJiraToIssue()`

### Step 3: issue-to-jira Capability

**Description**

**Parallel with Step 2.**

Implement the `pio_issue_to_jira` tool and `/pio-issue-to-jira` command. Reads a local issue file, extracts title and description body, and creates a Jira ticket via `acli jira workitem create`. The `--project` parameter is required unless a default exists in `.pio/jira-config.yaml`. Returns the created Jira key from the JSON response.

Uses `readIssue()` from `fs-utils.ts` to read the issue file, parses the markdown to extract title (first `# heading`) and body (remaining content). Uses `readJiraConfig()` to provide project default. Uses `runAcli()` to create the ticket.

**Acceptance Criteria**

- `npm run check` reports no errors
- `pio_issue_to_jira` is registered as a tool via `pi.registerTool()`
- `/pio-issue-to-jira` is registered as a command via `pi.registerCommand()`
- Returns an error when the issue slug doesn't correspond to an existing file
- Accepts `--project` as a required tool parameter (overridden by config default if absent)
- Reads defaults from `.pio/jira-config.yaml` when file exists and user omits explicit project
- Passes raw description text to `acli jira workitem create` (plain-text format)

**Files Affected**

- `src/capabilities/issue-to-jira.ts` — new file: tool, command, and setup function following create-issue.ts pattern
- `src/index.ts` — import and call `setupIssueToJira()`

## Notes

- **acli JSON structure is unverified:** The GOAL.md notes that actual field names may differ from assumed `summary`/`description` at top level. The implementer should check `acli jira workitem view --help` or run a test command if auth is available to confirm field names.
- **Jira ADF vs plain text:** Local issue markdown may not be valid Atlassian Document Format. The implementation passes raw text and lets `acli` handle conversion, or documents that complex formatting may not transfer perfectly.
- **No new dependencies:** All functionality uses existing deps (`js-yaml`, `typebox`) plus Node.js built-in `child_process`. No package.json changes needed.
- **Config file is optional:** `.pio/jira-config.yaml` must never block the command — if it doesn't exist, all fields fall back to requiring explicit user input where necessary (e.g., `--project` for issue-to-jira).
