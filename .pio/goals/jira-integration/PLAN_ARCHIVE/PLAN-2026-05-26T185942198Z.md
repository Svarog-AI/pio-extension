### Step 3: Create the pio-jira Skill

**Description:**

Create `src/skills/pio-jira/SKILL.md` ‚Äî a skill that instructs agents on performing all Jira operations via `acli` using the `bash` tool, combined with existing pio tools like `pio_create_issue`. This replaces the need for dedicated TypeScript capabilities (Steps 1‚Äì2) and the original Step 3 (`issue-to-jira` capability).

The skill must cover:

- **Auth status check:** Run `acli jira auth status` to verify authentication. If not authenticated, direct user to `acli jira auth login`.
- **Pull Jira ‚Üí local issue (jira-to-issue replacement):** Agent runs `acli jira workitem view KEY --json`, parses JSON output for `summary` and `description` fields, derives slug (`jira-<project>-<number>`), checks if `.pio/issues/<slug>.md` already exists, and calls `pio_create_issue` to create the file. If the file exists, warn without overwriting.
- **Push local issue ‚Üí Jira (issue-to-jira replacement):** Agent reads `.pio/issues/<slug>.md`, extracts title (first `# heading`) and body, resolves project key from `--project` parameter or `.pio/jira-config.yaml` (if it exists), then runs `acli jira workitem create --summary "..." --project "KEY" --type "Task" --description "..." --json`. Returns the created Jira key from JSON response.
- **Search with JQL:** Run `acli jira workitem search --jql "..." --json` for flexible querying. Document common patterns (e.g., `project = PROJ AND status = Open`).
- **Error handling:** If `which acli` fails, report helpful message about installing `acli`. If output contains "unauthorized", direct to `acli jira auth login`.
- **Config file convention:** Document `.pio/jira-config.yaml` format (`projectKey`, `defaultType`) as optional per-repo defaults for push operations.
- **Always use `--json`:** Emphasize that `--json` flag should always be used for programmatic output parsing.
- **Plain-text descriptions:** Note that local markdown may not transfer perfectly to Jira ADF ‚Äî pass raw text and let `acli` handle conversion.

Follow the existing skill structure conventions (see `src/skills/pio-git/SKILL.md` for format reference): YAML frontmatter with `name` and `description`, organized sections with progressive disclosure.

## Notes

- **Architectural pivot:** The original plan built typed TypeScript capabilities (`jira-utils`, `jira-to-issue`, `issue-to-jira`). The revision determined that agents already have everything needed via `bash` + existing tools (`pio_create_issue`). A skill is the right abstraction for flexible Jira interactions (search, updates, comments, transitions ‚Äî not just create/pull).
- **No new dependencies:** The skill-only approach requires zero new TypeScript files or package.json changes. The deliverable is a single markdown file that auto-discovers as a pi skill.
- **Skill auto-discovery:** Skills under `src/skills/*/SKILL.md` are auto-discovered by pi ‚Äî no manual registration needed. Verify the skill appears in `<available_skills>` after creation.
- **acli JSON structure remains unverified:** The GOAL.md notes that actual field names from `acli jira workitem view --json` may differ from assumed `summary`/`description`. The skill should instruct agents to inspect actual output and adapt if needed.
- **Steps 1‚Äì2 are historical anchors only:** The completed implementations are documented for context but will be deleted in Step 4. Do not attempt to preserve or reuse their code.
