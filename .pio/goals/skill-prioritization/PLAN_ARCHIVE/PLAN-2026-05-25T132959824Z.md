# Plan: Centralized capability-to-skill mapping

Consolidate skill loading into a single source of truth in `CapabilityConfig.skills`, replacing scattered `## Skill References` sections across prompt files with dynamic injection via `session-capability.ts`.

## Prerequisites

None.

## Steps

### Step 1: Add `skills` field to types

Add a `skills` configuration field to both `StaticCapabilityConfig` and `CapabilityConfig` in `src/types.ts`. The schema follows the structure defined in GOAL.md:

```typescript
interface CapabilitySkills {
  /** Skills forcefully injected into the prompt — full SKILL.md content is read at startup */
  mandatory?: string[];
  /** Skills listed as instructions for the LLM to load when conditions apply */
  recommended?: { name: string; condition: string }[];
}
```

`StaticCapabilityConfig.skills` uses this optional field. `CapabilityConfig.skills` also uses it — the runtime config shape mirrors the static one for skills (no callback resolution needed, skills are static per capability).

- [ ] `CapabilitySkills` interface is exported from `src/types.ts`
- [ ] Both `StaticCapabilityConfig` and `CapabilityConfig` include the optional `skills` field of type `CapabilitySkills`

**Files affected:**

Update `resolveCapabilityConfig()` in `src/capability-config.ts` so that when it constructs the runtime `CapabilityConfig`, it copies the `skills` field from the static config. This ensures the skills configuration flows from each capability module (`CAPABILITY_CONFIG`) into the session's runtime config entry (`pio-config`).

### Step 3: Implement skill injection logic in session-capability.ts

Modify `session-capability.ts` so that during `before_agent_start`, mandatory skills are force-injected and recommended skills are listed as instructions. This replaces the current static `_skill-loading.md` injection.

**Key implementation details:**
- Cache skills from `BeforeAgentStartEvent.systemPromptOptions.skills` to resolve skill name → `{ filePath, baseDir }` mapping. This provides filesystem paths for all registered skills (bundled + external) without coupling to internal pi APIs.
- Read each mandatory SKILL.md file using `readFileSync`, strip frontmatter using `stripFrontmatter` from `@earendil-works/pi-coding-agent`, and wrap in `<skill name="..." location="...">\nReferences are relative to <baseDir>.\n\n<body>\n</skill>` XML format — matching pi's own `_expandSkillCommand` pattern exactly.
- Generate a recommended skills section listing each skill name with its condition as `--- RECOMMENDED SKILLS ---` instructions.
- Construct the `--- SKILL LOADING INSTRUCTIONS ---` section dynamically: mandatory skill content blocks + recommended skill instructions. This replaces loading `_skill-loading.md` from disk.
- Global mandatory skills (`pio`, `ask-user`) are always injected on top of capability-specific skills. Read them once at session startup regardless of per-capability config.
- Delivery order in the custom message: (1) `--- PROJECT OVERVIEW ---`, (2) `--- SKILL LOADING INSTRUCTIONS ---` (dynamically generated), (3) `--- YOUR INSTRUCTIONS ---`.

**Acceptance criteria:**
- [ ] `npx tsc --noEmit` reports no errors
- [ ] Existing test suite passes with no regressions (`npm test`)
- [ ] `before_agent_start` generates the skill-loading section from capability config instead of reading `_skill-loading.md` from disk
- [ ] Mandatory skills are read, frontmatter stripped, and wrapped in `<skill>` XML tags following pi's expansion format
- [ ] Recommended skills appear as instruction-based listings with conditions
- [ ] Global mandatory skills (`pio`, `ask-user`) are always injected regardless of per-capability config
- [ ] Missing SKILL.md files are handled gracefully (logged warning, no crash)

**Files affected:**

### Step 6: Dynamic skill passing between sessions

Enable any capability to dynamically extend (not replace) the mandatory and recommended skills for downstream sessions. The mechanism flows through session params → config resolution → skill injection, allowing evolve-plan to recommend per-step skills via TASK.md frontmatter that execute-task picks up when launching.

**Mechanism:**
- Session params (`sessionParams` on `CapabilityConfig`) gain two optional fields: `dynamicMandatorySkills?: string[]` and `dynamicRecommendedSkills?: { name: string; condition: string }[]`. These extend the base capability config skills — they are merged (union), never replaced.
- `resolveCapabilityConfig()` in `capability-config.ts` merges `skills` from the static config with these dynamic params: mandatory arrays concatenated, recommended arrays concatenated (deduplicated by skill name).
- `session-capability.ts` picks up the merged list during `before_agent_start` — no new logic needed here beyond consuming whatever `CapabilityConfig.skills` contains.
- For the evolve-plan → execute-task use case: TASK.md gains an optional frontmatter field for recommended skills. When execute-task launches (via `next-task.ts` or direct tool), it reads TASK.md frontmatter, extracts skill recommendations, and passes them as `dynamicRecommendedSkills` in session params. This requires:
  - Adding a `TASK_FRONTMATTER_SCHEMA` to `frontmatter-schemas.ts` with optional `recommendedSkills?: { name: string; condition: string }[]`
  - Updating `next-task.ts` (or the execute-task launcher) to parse TASK.md frontmatter and inject dynamic skills into session params
- `evolve-plan.md` prompt instructs the spec writer to include recommended skills in TASK.md frontmatter when specific skills are needed for that step.

**Acceptance criteria:**
- [ ] `npx tsc --noEmit` reports no errors

## Notes

- **Bundled resource references within SKILL.md:** Skills like `pio-git` reference `[REFERENCE.md](REFERENCE.md)` for progressive disclosure. When injecting mandatory skill content, inject only the SKILL.md file itself. Bundled resources are referenced by path in the SKILL.md body — the LLM can follow them via `read` tool when needed. This preserves progressive disclosure while still guaranteeing the main skill is present.
- **Token budget:** Mandatory skill injection increases prompt size. The `test-driven-development` skill is substantial (~400 lines). This is accepted as a trade-off for guaranteed loading — recommended skills remain instruction-based to preserve token budget for situational use.
- **`_skill-loading.md` retention:** The file is retained on disk but no longer loaded by `session-capability.ts`. It serves as documentation of the old format and can be deleted in a future cleanup goal if desired. No step removes it — staying within scope.
- **External skill resolution:** `systemPromptOptions.skills` from `BeforeAgentStartEvent` provides the complete skill registry (bundled + external) with filesystem paths. This is used to resolve skill name → `{ filePath, baseDir }` without accessing pi internals or hardcoding paths.
- **Dynamic skill merging semantics:** Dynamic skills from session params extend the base config — they are never replacements. If a dynamic skill name duplicates a base skill, it is deduplicated (base entry wins for mandatory; first-seen wins for recommended). This ensures capability-level guarantees are always present.
