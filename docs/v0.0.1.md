# pio-extension v0.0.1 — Feature Overview

## What is pio?

pio is a **goal-driven project management workflow** for the [pi](https://github.com/earendil-works/pi-coding-agent) coding agent framework. It breaks complex work into structured sub-sessions with validation gates, prompt templates, and explicit output requirements — giving agents (and developers) a clear lifecycle to follow from idea to shipped code.

## Workflow Lifecycle

```
init → create-goal → create-plan → evolve-plan → execute-task → review-task → finalize-goal
                                                          ↗      ↖
                                                   revise-plan  (branch on divergence)
```

1. **Init** — Bootstrap the `.pio/` workspace directory.
2. **Create Goal** — Interview the user, produce `GOAL.md` in a goal workspace.
3. **Create Plan** — Research codebase, produce `PLAN.md` with numbered steps.
4. **Evolve Plan** — Generate `TASK.md` + `TEST.md` for the next incomplete step.
5. **Execute Task** — Implement a single step using test-driven development.
6. **Review Task** — Inspect implementation against spec, approve or reject with review notes.
7. **Revise Plan** — (Optional) Archive current plan, delete incomplete steps, rewrite plan for remaining work. Triggers when the specification writer detects significant divergence.
8. **Finalize Goal** — Read accumulated decisions across all completed steps, update `.pio/PROJECT/` documentation.

## Currently Supported Features

### Project Initialization

- **`pio_init` / `/pio-init`** — Create the `.pio/` directory structure (goals, issues, PROJECT, session-queue).

### Goal Management

- **`pio_create_goal` / `/pio-create-goal <name>`** — Create a goal workspace under `.pio/goals/<name>/`, queue a definition sub-session that writes `GOAL.md`.
- **`pio_delete_goal` / `/pio-delete-goal <name>`** — Remove a goal workspace entirely.
- **`pio_list_goals` / `/pio-list-goals`** — List all goals with current phase and last task.

### Issue Tracking

- **`pio_create_issue`** — Create an issue as a markdown file under `.pio/issues/<slug>.md`.
- **`pio_goal_from_issue`** — Convert an existing issue into a structured goal (queues a `create-goal` session).

### Planning

- **`pio_create_plan` / `/pio-create-plan <name>`** — Read `GOAL.md`, research the codebase, produce `PLAN.md` with numbered steps and acceptance criteria.
- **`pio_revise_plan` / `/pio-revise-plan <name>`** — Archive current `PLAN.md`, delete incomplete step folders, queue a fresh planning session for remaining work.

### Specification

- **`pio_evolve_plan` / `/pio-evolve-plan <name>`** — Generate `TASK.md` + `TEST.md` for the next incomplete step from `PLAN.md`. Places them in `S{NN}/` under the goal workspace.

### Implementation

- **`pio_execute_task` / `/pio-execute-task <name>`** — Implement a single plan step using test-driven development (write tests first, then code).
- **`pio_execute_plan` / `/pio-execute-plan <name>`** — Implement all steps from `PLAN.md` in a single session.

### Review

- **`pio_review_task` / `/pio-review-task <name>`** — Read `TASK.md`, `TEST.md`, `SUMMARY.md` and implementation files. Write `REVIEW.md` with categorized issues. Approve or reject the step.

### Project Context

- **`pio_create_project_context` / `/pio-project-context`** — Analyze project files and generate 7 documentation files under `.pio/PROJECT/` (tech stack, patterns, architecture decisions, etc.).

### Finalization

- **`pio_finalize_goal`** — Read accumulated decisions from completed steps, update `.pio/PROJECT/` documentation to reflect new knowledge.

### Task Queue

- **`/pio-next-task`** — Dequeue the next pending task from `.pio/session-queue/` and launch the appropriate sub-session.
- Per-goal task slots: only one task per goal runs at a time.

### Session Navigation

- **`/pio-parent`** — Switch back to the parent session from a sub-session.
- **`pio_mark_complete`** — Validate that expected output files exist for the current step, then mark it complete and auto-enqueue the next workflow task.

### Guardrails

- **File protection (readOnly / writeAllowlist)** — Sub-sessions can read any file but write only to their goal workspace unless explicitly allowed. Prevents accidental modifications outside the working area.
- **Turn guard** — Detects thinking-only turns (agent stuck in contemplation) and sends recovery prompts to unblock execution.

### Skills (discoverable by pi)

| Skill | Purpose |
|-------|---------|
| `pio` | pio workflow reference — tools, commands, lifecycle |
| `test-driven-development` | TDD methodology guide — red-green-refactor |
| `pio-project-knowledge` | Canonical knowledge of the 7 `.pio/PROJECT/*.md` files |
| `pio-planning` | Shared planning methodology — step structure, acceptance criteria, research process |

### Configuration

- **`~/.pi/pio-config.yaml`** — Per-capability model configuration (override which LLM handles which workflow phase).
